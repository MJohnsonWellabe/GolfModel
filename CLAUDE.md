# Claude Instructions

Before making any changes:

1. Read `docs/README.md` — it defines the governing documentation set and the
   precedence order (executive vision → design constitution → V2 roadmap →
   domain specs → implementation records → archive).
2. Read `docs/vision/00_EXECUTIVE_VISION.md` and
   `docs/vision/01_DESIGN_CONSTITUTION.md`.
3. Read the domain documents relevant to the work, and the current phase in
   `docs/roadmap/CLAUDE_PHASE_SEQUENCE.md` when following the V2 roadmap.
4. Respect `docs/technical/PERFORMANCE_AND_QUALITY_GATES.md` — performance is
   a release criterion, and analytics/persistence stay off critical input
   paths.
5. Before editing the generated dev courses (Red Hollow / Wild Prairie), the
   bounded-world renderer, or the terrain tests, read
   `docs/technical/COURSE_AND_BOUNDED_WORLD_FIELD_GUIDE.md` — those courses are
   GENERATED by `scripts/gen-new-courses.mjs` (never hand-edit the JSON), the
   vertical unit is ≈1.25 ft (not yards), and a wall of terrain gates enforces
   every promise. The field guide is the map of those traps.

Working principles:

6. Never sacrifice polish for speed.
7. Keep the game in a working state after every change.
8. Refactor when appropriate — never leave a file in worse condition than you
   found it.
9. Keep rendering separate from gameplay, UI separate from physics, Firebase
   isolated behind its modules, and configuration data-driven.
10. Update documentation if behavior changes.
11. Do not introduce placeholder implementations unless explicitly requested.
12. Prefer production-quality solutions over temporary fixes.
13. Dispose what you create: observers, scenes, textures, timers, listeners,
    and audio must not accumulate across rounds.
